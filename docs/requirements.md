# 项目需求：限制 macOS 耳机输出最大音量

## 1. 项目目标

开发一个仅在本机运行的 macOS 小工具：

- 持续监听当前默认音频输出设备。
- 如果当前输出为耳机类设备时，系统输出音量不得超过 20%。
- 如果音量被调高到 20% 以上，程序应立即将其压回 20%。

## 2. 当前实现基线

当前项目已经采用 Swift 实现，形式为命令行程序，并支持作为 LaunchAgent 在登录后自动运行。

当前实现的核心行为是：

- 监听默认输出设备变化。
- 监听输出音量变化。
- 通过设备名或输出源名称识别耳机类设备。
- 对耳机输出执行音量钳制。
- 在事件触发之外，额外用短周期轮询补充兜底。

后续如果让 AI 修改项目，应优先基于现有 Swift 方案迭代，而不是无理由重写技术栈。

## 3. 功能需求

### 3.1 输出设备识别

- 只处理“当前默认输出设备”。
- 当默认输出切换时，程序应自动切换监听目标。
- 耳机类设备可通过设备名、输出源名、关键字匹配等方式识别。
- 至少应覆盖：有线耳机、蓝牙耳机、AirPods。

### 3.2 音量限制

- 当且仅当当前输出属于耳机类设备时，执行音量上限限制。
- 上限固定为 20%。
- 如果当前音量小于等于 20%，不做额外干预。
- 如果当前音量大于 20%，应尽快压回 20%。
- 当用户连续快速调高音量时，程序仍应尽量稳定地把音量维持在 20% 以内。

### 3.3 运行方式

- 支持前台直接运行，便于调试。
- 支持安装为 LaunchAgent，便于登录后自动启动。
- 程序应长期运行，不依赖图形界面。

### 3.4 可观察性

- 程序启动时应输出当前监听的设备信息。
- 当发生音量压制时，应输出简明日志，便于排查是否生效。
- 自动运行模式下，应保留 stdout/stderr 日志文件。

## 4. 非功能要求

- 平台限定为 macOS。
- 优先保证稳定、简单、可长期运行。
- 不引入与目标无关的复杂架构。
- 尽量减少 CPU 占用，轮询只能作为兜底，不应过重。
- 修改应保持现有命令行使用方式和安装脚本可用。

## 5. 明确非目标

以下内容默认不在本项目范围内，除非我明确提出：

- 开发 GUI 界面。
- 支持 Windows 或 Linux。
- 支持复杂配置中心或云同步。
- 支持按不同应用分别限制音量。
- 将项目重写为其他语言或其他运行时。

## 6. 验收标准

满足以下条件即可认为需求达成：

1. 插入有线耳机或连接 AirPods 后运行程序。
2. 手动把系统音量调到 20% 以上。
3. 音量会被快速压回 20%。
4. 切换到非耳机输出时，程序不再主动限制音量。
5. 重新登录系统后，如已安装 LaunchAgent，程序会自动启动并继续生效。

## 7. 与 AI 协作时的约束

以后如果我让 AI 修改这个项目，默认按下面原则执行：

- 先理解现有实现，再改代码，不要先重构。
- 优先做小而准的修改，避免无关改动。
- 不要随意改动产品目标：核心目标始终是“仅在耳机输出时限制最大音量为 20”。
- 如果要调整耳机识别逻辑、轮询策略、监听策略，必须说明原因和可能副作用。
- 如果要新增依赖或改安装脚本，必须说明必要性。
- 除非我明确要求，否则不要增加 UI、配置文件或跨平台支持。

## 8. 常用命令

开发和验证时，优先使用以下命令：

```bash
swift build
swift run volume-guardian
./scripts/install-launch-agent.sh
./scripts/uninstall-launch-agent.sh
```
